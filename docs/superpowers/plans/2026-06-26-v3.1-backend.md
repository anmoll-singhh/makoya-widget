# v3.1 Backend Build — Master Coordination Plan

> **For agentic workers:** This is the MASTER plan. It defines the schema, contracts, lane decomposition, file ownership, and QA/security gates for the whole v3.1 backend. Each **wave** gets its own just-in-time task-level plan (TDD, bite-sized) authored by the coordinator before that wave is built. REQUIRED SUB-SKILL per lane: `superpowers:test-driven-development`. Execution model: one **coordinator** (orchestrates, owns shared files, runs gates, merges) + **subagents** per lane (each in its own git worktree/branch).

**Goal:** Make the backend ready for the `docs/makoya_v3.1.html` dashboard so the future frontend is pure integration against endpoints that already exist + are tested + security-reviewed.

**Architecture:** Additive, dependency-ordered waves. New tables/endpoints follow existing patterns exactly (timestamped SQL migrations, `owner_id`/org RLS, `SupabaseClient`-taking lib fns with snake↔camel mappers, Zod-gated never-500 routes, observability seam). No frontend is built — only DB + API + lib + tests + docs.

**Tech stack:** Next.js 15 / React 19, Supabase (Postgres + RLS), TypeScript, Zod, vitest, Upstash rate-limit, `@makoya/shared` canonical config.

## Global Constraints (verbatim from CLAUDE.md — every task inherits these)

- **One worktree = one branch = one agent.** Claim a lane on `docs/STATUS.md` agent board before editing. Never edit files outside your lane.
- **Shared files are coordinator-owned.** `packages/shared`, `apps/web/lib/shared` (generated — never hand-edit; regenerate with `npm run sync:shared`), `package.json`, `apps/web/lib/sites.ts`, `apps/web/lib/email`, `docs/STATUS.md`. If a lane needs one, it requests the coordinator; lanes never edit these directly.
- **Migrations are canonical in `supabase/migrations/`** with `YYYYMMDDHHMMSS_name.sql` names. Each lane owns ONE migration file with a coordinator-assigned timestamp (so files never collide). RLS enabled on every new table.
- **QA gate around every merge:** `npm run ci` green on up-to-date base + branch (QA-before) AND re-run on `main` post-merge (QA-after). Never merge on red/stale base.
- **Security posture:** public endpoints treat all input as hostile — Zod at the boundary, rate-limited (Upstash, fail-open), generic errors (no oracles), never-500 contract, service-role never reaches client.
- **Compliance copy:** no "WCAG/ADA compliant" or "guaranteed accessible" claims. Tools/preferences + *effort*, never certification.
- **Observability:** all analytics/error calls route through `apps/web/lib/observability.ts`. No raw `console` at feature call sites.
- **Deploy only from clean `main`** (`vercel --prod` from `apps/web`). Coordinator does deploys, never lanes.
- **Migration application is founder-gated.** Lanes WRITE migration files + validate on a throwaway Supabase dev branch (`mcp__supabase__create_branch` → `apply_migration` → tests → `delete_branch`). **Applying to the prod project is a separate, explicit, founder-authorized step the coordinator does last.**

## Wave decomposition (dependency-ordered) + lane ownership

Each lane = one worktree + branch + subagent. Coordinator dispatches the lanes in a wave **in parallel** (they have disjoint file ownership), gates each, merges, then opens the next wave. Branch naming: `feat/v31-<lane>`.

### WAVE 1 — independent quick wins (no tenancy dependency)
Parallel-safe; disjoint files.

- **Lane 1A — Heartbeat / install-verify** (`feat/v31-heartbeat`)
  - Owns: `supabase/migrations/<ts>_widget_heartbeats.sql`, `apps/web/lib/heartbeat.ts` (+test), `apps/web/app/api/heartbeat/route.ts`, `apps/web/app/api/sites/[id]/install-status/route.ts` (+tests).
  - Schema: `widget_heartbeats(site_id uuid pk → sites, first_seen_at timestamptz, last_seen_at timestamptz, ping_count bigint default 0, last_url text)`; `widget_uptime_days(site_id uuid → sites, day date, pings int, primary key(site_id,day))`. RLS: owner-reads via sites subquery; **writes service-role only** (the public `/api/heartbeat` uses service client — widget is unauthenticated).
  - `/api/heartbeat`: public, Zod `{siteId, token?, url?}`, rate-limited (Upstash, namespace `heartbeat`, fail-open), never-500, upserts heartbeat + increments daily. Token grace identical to config route.
  - `deriveInstallStatus(siteId)` → `active|monitoring|action_needed|not_installed` from `last_seen_at` recency (+ latest scan score for action_needed). Pure function, unit-tested.

- **Lane 1B — Widget usage analytics** (`feat/v31-analytics`)
  - Owns: `supabase/migrations/<ts>_widget_events.sql`, `apps/web/lib/analytics.ts` (+test), `apps/web/app/api/widget-events/route.ts`, `apps/web/app/api/sites/[id]/analytics/route.ts` (+tests).
  - Schema: `widget_events(id uuid pk, site_id uuid → sites, event text check(event in ('open','feature_activated')), feature_key text, occurred_at timestamptz default now())`; rollup `widget_event_daily(site_id, day date, event text, feature_key text, count int, primary key(site_id,day,event,feature_key))`. RLS owner-read, service-role write.
  - `/api/widget-events`: public, Zod-validated **batch** `{siteId, token?, events:[{event,featureKey?,ts?}]}` (cap N), rate-limited, fail-open, never-500. Increments daily rollup.
  - `getWidgetAnalytics(siteId, range)` → opens, feature-activation counts, most-used, opens-over-time series.
  - **NOTE (coordinator-owned dependency):** the widget bundle must emit these events. That is a *separate* `packages/widget` change the coordinator schedules after the endpoint lands (keeps lane disjoint from shared widget code).

Coordinator resolves the shared read of issues/heartbeat via stable lib interfaces from 1A/2A.

## Per-lane gate pipeline (every lane, no exceptions)

Coordinator runs this for each lane before merge:

1. **PLAN** — coordinator writes the lane's bite-sized TDD task list (JIT, just before dispatch) into `docs/superpowers/plans/lanes/<lane>.md`.
2. **CODE** — build subagent implements task-by-task via `superpowers:test-driven-development` in its worktree. Migration validated on a Supabase dev branch (create → apply → delete), never prod.
3. **QA** — a fresh read-only verification subagent runs `npm run ci` (sync:shared + typecheck web+widget + vitest) on the lane branch rebased on latest `main`; confirms new tests exist and pass, drift gate green. Reports PASS/FAIL with output.
4. **SECURITY** — a fresh security-review subagent audits the diff: RLS correctness (no cross-tenant read), Zod at every public boundary, rate-limit + fail-open + never-500 on public routes, no service-key leak to client, generic errors (no SSRF/enumeration oracle), compliance-copy check. Reports findings.
5. **MERGE** — only on QA-green + security-clean + up-to-date base. Coordinator merges, runs QA-after on `main`, updates STATUS.md agent board.
